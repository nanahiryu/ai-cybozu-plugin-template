# プロジェクト・ビジョン: AI ネイティブな kintone 開発

## 概要

本プロジェクトは、**Cursor** や **Claude Code** といった AI 開発ツールを前提とした、次世代の kintone プラグイン開発基盤（エコシステム）です。
単なるテンプレートや CLI ツールではなく、AI との開発フロー（対話、設計、実装、テスト）を最適化するための「コンテキスト」と「ガイド」を提供します。

## コア・バリュー (Core Values)

### 1. AI ネイティブなワークフロー (AI-Native Workflow)

人間がコードを書くのではなく、AI にコードを書かせることを前提とします。
開発者は「指示（Prompt）」「設計（Design）」「レビュー（Review）」に集中し、実装の詳細は AI が担います。
`/plan` コマンドなどの AI 向けトリガーを通じて、開発プロセスをリードします。

### 2. テストファーストと堅牢性 (Test-First & Robustness)

AI 生成コードの品質を担保するため、テスト駆動開発（TDD）を強制します。
まずテスト（E2E, Integration, Unit）を定義・実装し、それをパスするように AI に実装させることで、手戻りのない堅牢な開発を実現します。

### 3. Issue 駆動開発 (Issue-Driven Development)

漠然とした指示でコーディングを開始せず、必ず「実装計画」を立て、GitHub Issue としてタスク化してから着手します。
これにより、コンテキストの汚染を防ぎ、AI が迷子になるのを防ぎます。

### 4. クリーンアーキテクチャと可読性

**Simplified Hexagonal Architecture** を採用し、コードの責務を明確に分離します。
過度な複雑さを避けつつ、テスト容易性を確保する 4 層構造を基本とします。

1. **Domain**: 純粋なビジネスロジック（kintone API 非依存）。単体テストの中心。
2. **Infra**: kintone API や外部 API との通信を行うアダプター。
3. **UI**: React コンポーネント。表示とユーザー操作を担当。
4. **Handler**: kintone イベントの受け口。各層をつなぐエントリーポイント。

AI が生成するコードであってもブラックボックス化せず、人間が理解しやすく、長期的に保守可能な構造を維持します。

## 提供するもの

- **インタラクティブなガイド**: `/plan` などで発動する、工程ごとの最適なプロンプト定義。
- **開発の土台**: テスト環境（Playwright, Vitest）やビルド環境が完備された土台。
- **コンテキスト認識**: kintone のアプリ構成や API 仕様を AI に正しく理解させるための仕組み。
